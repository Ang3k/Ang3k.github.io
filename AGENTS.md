# Briefing permanente do portfólio

## Objetivo do projeto

Este repositório contém o novo portfólio profissional de Angel Mansilla. O site
deve apresentar sua trajetória, experiência, competências e projetos nas áreas de
Machine Learning, Ciência de Dados e Engenharia de Machine Learning.

O novo site substituirá o portfólio atual do Wix, preservando sua identidade visual
e sua forma de apresentar estudos de caso, mas com conteúdo profissional atualizado,
melhor organização, melhor desempenho e responsividade real.

## Resultado esperado

- Um portfólio estático, rápido e publicável no GitHub Pages.
- Identidade visual reconhecível em relação ao site atual do Wix.
- Posicionamento profissional atualizado, com foco maior em Machine Learning
  Engineering e aplicações práticas de modelos.
- Projetos apresentados com contexto, metodologia, ferramentas, resultados,
  limitações e links relevantes.
- Experiência consistente em desktop, tablet e celular.
- Conteúdo fácil de atualizar sem depender do Wix.

## Direção visual

O site deve preservar os elementos mais marcantes do Wix:

- Estética minimalista e editorial.
- Fundo predominantemente branco.
- Paleta principal em preto, branco e tons de cinza.
- Nome “Angel Mansilla” em grande escala no cabeçalho.
- Linha divisória fina sob o nome.
- Navegação centralizada, simples e discreta.
- Títulos altos e leves, inspirados na fonte Oswald ExtraLight.
- Uso pontual de uma fonte serifada, inspirada na Playfair Display, para textos de
  apresentação.
- Grade de projetos com imagens quadradas ou proporcionais e títulos objetivos.
- Páginas de projeto com grandes imagens de abertura, seções numeradas, gráficos,
  trechos de código e navegação entre partes.
- Rodapé simples com e-mail, LinkedIn e GitHub.

A referência deve orientar o novo design, não ser copiada com todas as suas
limitações. O site novo não deve conter propaganda, marca ou textos do Wix.

### Comportamento visual desejado

- Tipografia fluida, sem tamanhos fixos que estourem em telas pequenas.
- Largura de leitura confortável para estudos de caso.
- Cards responsivos: três colunas em telas largas, duas em tablets quando couber e
  uma em celulares.
- Imagens responsivas, sem deformação ou rolagem horizontal.
- Hierarquia visual clara entre contexto, processo, resultado e conclusão.
- Animações discretas e opcionais, respeitando `prefers-reduced-motion`.
- Contraste adequado e estados de foco visíveis.

#### Dengue Sense Classifier

- Classificação de casos de dengue e chikungunya com dados públicos do SINAN.
- Pipeline com 107 features.
- Modelos MLP, XGBoost e LightGBM combinados em ensemble.
- Aplicação com frontend em React e TypeScript e API em FastAPI.
- Processamento e auditoria de dados entre 2014 e 2021.
- Aproximadamente 11,4 milhões de registros brutos baixados.
- Teste temporal final em 940.304 registros de 2021.
- Ensemble publicado com recall de 0,873 e ROC-AUC de 0,829 no teste descrito.
- O resultado é um score de classificação, não diagnóstico ou probabilidade clínica.
- Repositório:
  <https://github.com/Ang3k/Dengue-Ensembler-Classifier>

Esse projeto possui backend. O GitHub Pages pode hospedar o estudo de caso e o
frontend estático, mas a API precisa ser hospedada separadamente para uma

- Repositório:
  <https://github.com/Ang3k/CPU-Temperature-Anomaly-Detector>

## Compatibilidade com GitHub Pages

Toda implementação deve gerar um site estático compatível com GitHub Pages.

### Restrições obrigatórias

- Não depender de servidor de aplicação próprio durante uma visita.
- Não depender de processos persistentes no ambiente do GitHub Pages.
- Não expor chaves, tokens ou outros segredos no JavaScript entregue ao navegador.
- Entregar HTML, CSS, JavaScript e recursos estáticos.
- Se houver framework ou gerador, o build final deve ser completamente estático.
- O build deve poder ser executado por GitHub Actions.
- Os caminhos devem funcionar no subdiretório do repositório, sem assumir publicação
  na raiz de um domínio.
- Rotas não podem depender de fallback de servidor que o GitHub Pages não forneça.
- Aplicações FastAPI, Streamlit ou outros backends devem ser hospedadas fora do
  GitHub Pages e apenas vinculadas ou consumidas de forma segura.
- Dashboards externos, como Power BI, podem ser incorporados ou vinculados desde que
  funcionem no navegador e não exijam segredos no cliente.

## Requisitos de qualidade

### Responsividade

- Adotar abordagem mobile-first.
- Não permitir rolagem horizontal em larguras comuns.
- Testar pelo menos em 320, 375, 390, 768, 1024 e 1440 pixels.
- O Wix atual apresenta cerca de 590 pixels de overflow horizontal em uma viewport
  de 390 pixels; esse comportamento não deve ser reproduzido.

### Acessibilidade

- HTML semântico.
- Navegação completa por teclado.
- Link de pular para o conteúdo.
- Contraste adequado.
- Textos alternativos descritivos para imagens relevantes.
- Imagens decorativas com alternativa vazia.
- Títulos em ordem hierárquica.
- Gráficos acompanhados de resumo textual.
- Respeito às preferências de redução de movimento.

### Desempenho

- Carregamento preguiçoso de imagens fora da primeira dobra.
- Imagens otimizadas e dimensionadas.
- Pouco JavaScript no carregamento inicial.
- Fontes com estratégias que evitem bloqueio e mudanças bruscas de layout.
- Nenhuma dependência pesada sem justificativa.

### SEO e compartilhamento

- Título e descrição únicos em todas as páginas.
- URL e nome de arquivo legíveis.
- Metadados Open Graph.
- Imagem de compartilhamento.
- Sitemap e `robots.txt`.
- Canonical coerente com a URL final.
- Dados estruturados apropriados para pessoa, perfil e projetos, quando aplicável.

### Conteúdo

- Português como idioma principal, salvo decisão posterior.
- Evitar a mistura não intencional de títulos em inglês com textos em português.
- Revisar ortografia, concordância e clareza dos textos migrados.
- Não inventar métricas, cargos, datas ou resultados.
- Informar limitações dos modelos e contexto das métricas.
- Não apresentar ferramentas de saúde como diagnóstico.

## Problemas do Wix que não devem ser reproduzidos

- Layout fixo e quebrado no celular.
- Propaganda e marca Wix.
- Conteúdo profissional desatualizado.
- Mistura inconsistente de português e inglês.
- Páginas com títulos internos como “backup”, “cópia” ou “protótipo”.
- Quarto projeto completo escondido da grade principal.
- Textos alternativos genéricos como `image.png`.
- Páginas extensas sem navegação interna ou resumo.
- Imagens muito grandes servidas em dimensões pequenas.
- Ausência de descrição SEO na maioria das páginas.
- Dependência do formulário e dos componentes proprietários do Wix.
